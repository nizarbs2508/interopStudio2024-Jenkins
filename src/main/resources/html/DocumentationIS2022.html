INTEROP STUDIO 2024


1.  Presentation generale

Interop Studio est un outil interne à léquipe Intéropérabilité de la DEI.

Il est en évolution permanente et à vocation à agréger des modules utilitaires destinés à lensemble de léquipe amenés à travailler sur loutillage Art Décor, Gazelle, ou encore sur la génération dexemples.

Interop Studio a été développé en Java sur loutil libre Eclipse.

 

 

2.  Module de cross-validation

2.1.    Définition de la cross-validation

 

La cross-validation consiste à recouper les informations présentes dans le document « metadata » dun fichier IHE_XDM.ZIP avec celles présentes dans lun des document CDA quil contient.

Interop Studio propose un module de cross-validation permettant :

-       une cross-validation complète où lensemble des règles sont évaluées et un rapport global produit

-       une cross-validation atomique où lutilisateur choisit dans une liste la règle à évaluer

 

2.2.    Chargement des fichiers CDA et METADATA

Le module de cross-validation dInterop Studio est accessible dans la fenêtre principale de lapplication :

Il est nécessaire , au préalable, de charger le CDA et le METADATA dans le logiciel, au moyen des boutons de sélection de fichiers ci-dessous :

A noter que si vous avez déjà utilisé Interop Studio, les chemins des fichiers préalablement chargés auront été mémorisés et seront  automatiquement rechargés au lancement de lapplication 

 

2.3.    Cross-validation complète

Une fois que lon a procédé au chargement des fichiers CDA et METADATA à valider, la cross-validation est lancée en cliquant sur le bouton Cross-Valider les fichiers CDA et META du groupe Fonction XDM

 

La cross-validation complète est alors lancée 

Pour les règles qui sont validées, le rapport affiche lidentifiant de la règle ainsi que les valeurs renvoyées par lévaluateur de règles XPath :

-------------------------------------------

****> Règle n°40 ITI-32_36_serviceStartTime

      CDA:{29/10/2019 10:17:00} == META:{29/10/2019 10:17:00}

----------------------

-------------------------------------------

****> Règle n°41 ITI-32_37_serviceStopTime

      CDA:{!!!{Chemin Xpath introuvable!}} == META:{!!!{Chemin Xpath introuvable!}}

----------------------

 

 

Dans les logs derreur, des informations complémentaires sont affichées pour permettre à lutilisateur de corriger rapidement son problème :

Lidentifiant de la règle :

****> Règle n°19 ITI-32_19_authorInstitution

 

Les valeurs renvoyées de part et dautre (CDA et METADATA) par lévaluateur de règles XPath :

CDA:{Centre de soins du Belvédère^^^^^&1.2.250.1.71.4.2.2&ISO^IDNST^^^1120456789} <> META:{Centre de soins du BelvédèreXXX^^^^^&1.2.250.1.71.4.2.2&ISO^IDNST^^^1120456789}

 

Une courte description de la règle :

Description de la règle:

3.  Fonctions de contrôle du cda

Interop Studio propose plusieurs fonctions de contrôle et de correction du CDA.

3.1.    Détection et correction des UUID invalides

Ce module contrôle l'ensemble des UUID dans l'attribut id/@root des sections et entrées et les corrige si nécessaire en générant un UUID valide.

Cette fonctionnalité sapplique sur le document CDA chargé dans la fenêtre principale 

Le module sera exécuté en appuyant sur le bouton « Correction des UUID des éléments <id> » qui se trouve dans le menu « Contrôles du CDA » de la fenêtre principale:

Les corrections sont effectuées directement sur le document CDA. Aucune autre action nest nécessaire.

3.2.    Calcul du Hash

Le bouton Calcul du hash avec canonisation préalable situé dans le menu « Contrôles du CDA » de la fenêtre principale effectue un calcul du hash du CDA chargé dans loutil Interop Studio 

La valeur du hash saffiche dans le cadre blanc de bienvenue de l'application

Lalgorithme utilisé pour le calcul du hash est le SHA1 de JAVA 

3.3.    Contrôle des codes Bio Loinc du CDA

Interop Studio permet un contrôle des codes Bio Loinc présents dans le CDA via le bouton « Contrôle des codes Bio Loinc du CDAC » du menu « Contrôles du CDA » de la fenêtre principale.

Le module contrôle les codes LOINC présents dans les éléments //*:observation/*:code[@codeSystem='2.16.840.1.113883.6.1']/@code du CDA.

Les résultats saffichent dans la fenêtre de bienvenue de l'application.

Le fichier utilisé pour le contrôle des codes LOINC est intégré dans l'API Interop Studio.

Le nom du fichier est paramétré dans le fichier « Loinc.csv » intégré dans l'application :

4.  Module de recherche XPATH dans un repertoire de CDA

Ce module permet dévaluer une expression Xpath dans lensemble des documents CDA dun répertoire donné.

Il permet, par exemple, de rechercher une erreur donnée dans un ensemble de fichier CDA, ou encore de recenser les CDA qui utilisent lélément priorityNumber etc

Ce module est accessible via le bouton « Module de recherche Xpath dans un répertoire de CDA » du menu « Recherche XPATH » de la fenêtre principale.

Le module mémorise le dernier répertoire de recherche utilisé, ainsi que la dernière expression Xpath utilisée.

Ces informations sont stockées dans le fichier interopStudio.ini qui se trouve dans le répertoire de lapplication, dans les éléments [MEMORY]LAST-PATH-USED et [DIAGNOSTIC]LAST-REQUEST

4.1.    Utilisation du module

Lutilisation de ce module nécessite deffecter les opérations suivantes :

-       sélection du répertoire dans le cas où le répertoire mémorisé ne convient pas.

-       saisie dune expression Xpath booléenne à tester dans lensemble des fichiers XML du CDA.

Lexpression Xpath peut ainsi être saisie directement dans la textbox bleue :

ou bien sélectionnée dans la bibliothèque de tests favoris placée au dessus :

-       Choix du type de résultats à afficher :

-       Lancer la recherche avec le bouton « Loupe » situé au milieu à droite de la fenêtre :

4.2.    Exploitation des résultats

Les résultats saffichent sous forme de chemin complets dans la liste en bas de lécran.

En double cliquant sur une ligne de la liste, le fichier sélectionné va souvrir dans léditeur par défaut paramétré dans windows pour les fichiers xml.
Lutilisateur peut ainsi examiner ou corriger les CDA un par un depuis Interop Studio.

5.2.    Validation de masse

Licône  symbolise la validation de masse dun répertoire entier de fichiers CDA.

Après sélection dun répertoire contenant les CDAs dans sa racine, la validation est lancée séquentiellement pour chacun des CDA.

Les fichiers validés sont copiés dans un répertoire « --- VALIDATION-FICHIERS VALIDES--- »

Les fichiers en échec (Validation échouée) sont copiés dans le répertoire « --- VALIDATION_CDA_FICHIERS_INVALIDES --- »

Les fichiers en échec pour cause de time out sont copiés dans le répertoire « --- VALIDATION_CDA_FICHIERS_INVALIDES --- \TimeOut »

La synthèse des opération effectuées est affichée dans la console gauche de l'API.

6.  Module de génération de IHE_XDM

6.1.    Accès au module IHE XDM

Laccès au module de génération IHE XDM se fait depuis la fenêtre principale via Générer un fichier de métadonnées XDM du menu Fonctions XDM 

6.2.    Génération darchives en masse

Néanmoins, il possible, depuis la fenêtre principale, de générer en masse des archives XDM à partir dun répertoire contenant un ensemble de fichiers CDA.

En cliquant sur Générer des archives XDM pour l'ensemble des CDA d'un répertoire du menu Fonctions XDM :

il vous sera proposé de sélectionner le répertoire dans lequel se trouve vos fichiers CDA.

La génération darchives correspondant aux CDAs du répertoire se fera selon le processus suivant :

-       Parcours des CDA du répertoire

-       Génération du fichier Metadata correspondant au CDA en cours.

-       Validation en ligne du Metadata et Cross-Validation via les APIs de lANS. Si lune de ces validations renvoie une erreur ou que lAPI est indisponible, la génération de cette archive est annulée. Le fichier CDA est copié dans un sous-répertoire appelé : « --- CDA_INVALIDES ---».

-       Si les validations sont positives, un sous-répertoire portant le nom du fichier CDA est crée, qui contiendra larchive IHE_XDM.ZIP, ainsi qun sous-répertoire appelé « Contenu du ZIP » dans lequel on trouvera larchive décompréssée.

-       Le résultat de ce processus daffichera dans la console à gauche pour chacun des CDAs.
