INTEROP STUDIO 2024


1.  GENERAL PRESENTATION

IInterop Studio is an internal tool of the DEI Interoperability team.

It is in constant evolution and aims to aggregate utility modules intended for the entire team working on Art Décor and Gazelle tools, or even on the generation of examples.

Interop Studio was developed in Java on the free tool Eclipse.

 

 

2.  CROSS-VALIDATION MODULE

2.1.    Definition of cross-validation

 

Cross-validation consists of cross-checking the information present in the 'metadata' document of an IHE_XDM.ZIP file with that present in one of the CDA documents it contains.

Interop Studio offers a cross-validation module allowing:

-       a complete cross-validation where all the rules are evaluated and an overall report produced

-       an atomic cross-validation where the user chooses the rule to evaluate from a list

3.  CDA CONTROL FUNCTIONS

Interop Studio provides several CDA monitoring and correction functions.

3.1.    Detecting and fixing invalid UUIDs

This module checks all UUIDs in the id/@root attribute of sections and entries and corrects them if necessary by generating a valid UUID.

This functionality applies to the CDA document loaded in the main window 

The module will be executed by pressing the “Fix UUIDs of elements” button which is located in the “CDA Controls” menu of the main window:

Corrections are made directly on the CDA document. No further action is necessary.

3.2.    Hash calculation

The Calculate hash with prior canonicalization button located in the “CDA Controls” menu of the main window performs a calculation of the hash of the CDA loaded in the Interop Studio tool 

The hash value is displayed in the white welcome frame of the application

The algorithm used for calculating the hash is JAVA's SHA1  

3.3.    Control of Bio Loinc from CDA codes

Interop Studio allows control of the Bio Loinc codes present in the CDA via the “Control of Bio Loinc CDAC codes” button in the “CDA Controls” menu in the main window.

The module controls the LOINC codes present in the //*:observation/*:code[@codeSystem='2.16.840.1.113883.6.1']/@code elements of the CDA.

The results are displayed in the application's welcome window.

The file used for LOINC code control is integrated into the Interop Studio API.

The file name is set in the “Loinc.csv” file integrated into the application :

4.  XPATH SEARCH MODULE IN A CDA DIRECTORY

This module allows you to evaluate an Xpath expression in all CDA documents in a given directory.

It allows, for example, to search for a given error in a set of CDA files, or to list the CDAs which use the priorityNumber element etc...

This module is accessible via the “Xpath search module in a CDA directory” button in the “XPATH search” menu in the main window.

The module remembers the last search directory used, as well as the last Xpath expression used.

This information is stored in the interopStudio.ini file which is located in the application directory, in the [MEMORY]LAST-PATH-USED and [DIAGNOSTIC]LAST-REQUEST elements

4.1.    Using the module

Using this module requires performing the following operations :

-       selection of the directory in case the stored directory is not suitable.

-       entry of a Boolean Xpath expression to test in all CDA XML files.

The Xpath expression can thus be entered directly into the blue textbox :

or selected from the favorite test library placed above :

-       Choice of the type of results to display :

-       Start the search with the “Magnifying glass” button located in the middle right of the window :

4.2.    Exploitation of results

The results are displayed as full paths in the list at the bottom of the screen.

By double clicking on a line in the list, the selected file will open in the default editor configured in Windows for xml files.
The user can thus examine or correct the CDAs one by one from Interop Studio.

5.2.    Mass validation

The icon symbolizes the mass validation of an entire directory of CDA files.

After selecting a directory containing the CDAs in its root, validation is launched sequentially for each of the CDAs.

The validated files are copied to a directory “---VALIDATION-VALID FILES---”

Failed files (Validation failed) are copied to the “---VALIDATION_CDA_FICHIERS_INVALIDES ---” directory.

Files that failed due to timeout are copied to the “---VALIDATION_CDA_FICHIERS_INVALIDES ---\TimeOut” directory.

The summary of the operations carried out is displayed in the left console of the API.

6.  IHE_XDM GENERATION MODULE

6.1.    Access to the IHE XDM module

Access to the IHE XDM generation module is done from the main window via Generate an XDM metadata file from the XDM Functions menu  

6.2.    Mass archive generation

However, it is possible, from the main window, to mass generate XDM archives from a directory containing a set of CDA files.

By clicking on Generate XDM archives for all CDAs in a directory in the XDM Functions menu :

You will be asked to select the directory where your CDA files are located.

The generation of archives corresponding to the CDAs in the directory will be done according to the following process :

-       Browse the CDAs in the directory

-       Generation of the Metadata file corresponding to the current CDA.

-       Online validation of Metadata and Cross-Validation via the ANS APIs. If one of these validations returns an error or the API is unavailable, the generation of this archive is canceled. The CDA file is copied to a subdirectory called: “---CDA_INVALIDES ---”.

-       If the validations are positive, a subdirectory bearing the name of the CDA file is created, which will contain the IHE_XDM.ZIP archive, as well as a subdirectory called “ZIP Content” in which we will find the decompressed archive .

-       The result of this process will be displayed in the console on the left for each of the CDAs.
